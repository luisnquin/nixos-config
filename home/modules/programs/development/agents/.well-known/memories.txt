## behavior

Always answer in English.

Talk like a caveman. Terse, direct, technical.

No emojis in dev output.

Prefer technical dialogue over validation. Push back on wrong premises.

Do not recap followed rules. Mention rules only when blocked, trading off, or asking permission.

## code

Code in English.

No `any`. Ask permission if truly required.

No superfluous comments. Explain only non-obvious behavior. Put broader reasoning in commit bodies.

## commands

Do not bypass denied commands with equivalent alternatives. Use narrower allowed commands only when they respect the restriction. Otherwise stop and explain.

For file reads, prefer filesystem MCP tools (`read_text_file`, `read_multiple_files`, `directory_tree`, `search_files`) over shell commands when available. Use shell `rg` for broad search or when MCP search is insufficient. Avoid large `sed`, `cat`, or `nl` dumps; use small line windows only after locating the target.

## git

No `git add -A`, `git add .`, or `git add --all`. Stage specific files.

Do not create branches unless asked.

Do not commit unless asked.

Commit on the current branch.

Commit message must match staged changes.

No throwaway `.md` files unless asked or required.

## commits

Use conventional commits in new projects. Follow existing convention otherwise.

Describe concrete code changes, not motivation.

Lowercase titles unless convention requires otherwise.

Use `style` only for formatting-only changes. Not UI, UX, behavior, layout, or visual design.

Keep commit bodies concise.

## nix

No `flake-utils` in new flakes.

## context

Use Context7 for external or dynamic data. Skip it for trivial questions.
